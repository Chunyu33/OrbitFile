# OrbitFile 后端模块说明

## 架构总览

```
src-tauri/src/
│
├── main.rs                 # 程序入口（6 行）
├── lib.rs                  # 模块枢纽：声明所有子模块 + run() 注册 Tauri 命令
│
├── models.rs               # 【基础层】共享数据结构（所有模块的上游依赖）
├── utils.rs                # 【基础层】跨模块共享的文件系统工具函数
│
├── system/                 # 【系统接口层】封装 Windows OS 信息查询
│   ├── disk_usage.rs       #   磁盘容量/使用率扫描（sysinfo）
│   └── icon.rs             #   从 .exe/.dll 提取图标 → Base64 PNG（Win32 API）
│
├── storage/                # 【存储层】数据目录与迁移历史的持久化读写
│   ├── data_dir.rs         #   数据目录配置（指针文件、自定义文件夹 CRUD）
│   └── history.rs          #   迁移历史 CRUD、幽灵链接清理、统计、导入导出、应用还原
│
├── folder_manager/         # 【文件夹管理层】系统/应用数据/自定义文件夹的迁移
│   └── mod.rs              #   大文件夹发现（dirs + detector）+ 迁移/恢复 + 模板管理
│
└── app_manager/            # 【应用管理层】Windows 已安装应用的全生命周期管理
    ├── scanner.rs          #   注册表扫描（HKLM/HKCU × 32/64-bit）+ 文件系统补充扫描
    ├── migration.rs        #   迁移引擎（空间检查 → 复制 → 校验 → Junction → 清理备份）
    ├── uninstaller.rs      #   卸载引擎（原始命令执行 → 注册表确认 → 残留扫描 → 安全清理）
    ├── detector.rs         #   特殊目录检测（聊天应用数据路径动态发现）
    ├── log_macros.rs       #   统一日志宏（orbit_log! / log_warn!）
    └── mod.rs              #   模块声明
```

## 模块依赖关系

```
models.rs  ←────────────────────────────────────────── 所有模块
utils.rs   ←────────────────── storage/  folder_manager/  app_manager/
    ↑                              ↑           ↑               ↑
    │                              │           │               │
system/                      storage/    folder_manager/  app_manager/
(独立,仅依赖    (data_dir ← history)   (依赖 storage +        (内部互调)
 models)                        app_manager)
```

- **`models.rs`** 和 **`utils.rs`** 是零上游依赖的基础模块，被所有其他模块引用
- **`system/`** 独立于其他模块，仅依赖 `models`
- **`storage/history.rs`** 依赖同级的 `data_dir.rs`
- **`folder_manager/`** 依赖 `storage/`（读历史）和 `app_manager/`（迁移引擎、detector）
- **`app_manager/`** 内部：`migration.rs` 依赖 `storage/history.rs`（写迁移记录）；`scanner.rs` 依赖 `system/icon.rs`（提取图标）

## 各模块详细说明

### 1. `lib.rs` — 模块枢纽（155 行）

**职责**: 声明所有子模块，注册 31 个 Tauri 命令，启动应用。

命令分三类：
- **直接转发**: 10 个命令（get_installed_apps、check_process_locks 等）→ 委托给 `app_manager/`
- **带状态注入**: 3 个命令（migrate_app、migrate_special_folder、cancel_migration）→ 注入 `MigrationState` 取消标志
- **模块路径直连**: 18 个命令（disk、data_dir、folder、history）→ `generate_handler![]` 中直接引用子模块函数

### 2. `models.rs` — 共享数据结构（281 行）

**职责**: 集中定义所有前后端共享的类型，零业务逻辑。

| 结构体 | 用途 |
|--------|------|
| `MigrationState` | Tauri 托管状态，持有取消标志 `Arc<AtomicBool>` |
| `InstalledApp` | 已安装应用（名称、路径、图标 Base64、发布商） |
| `DiskUsage` | 磁盘容量/可用/使用率 |
| `LargeFolder` / `LargeFolderType` | 大文件夹信息 + 类型枚举（System/AppData/Custom） |
| `LargeFolderSizeEvent` / `MigrationCompleteEvent` / `RestoreCompleteEvent` | 后台任务完成后的前端事件 |
| `DataDirConfig` / `CustomFolderEntry` | 数据目录配置 + 自定义文件夹条目 |
| `AppDataTemplate` | 应用数据模板（定义监控哪些应用的数据目录） |
| `MigrationResult` / `ProcessLockResult` | 迁移结果 + 进程占用检测结果 |
| `MigrationRecord` / `MigrationRecordType` / `HistoryStorage` | 迁移历史记录 + 类型枚举 + 持久化容器 |
| `LinkStatusResult` | Junction 链接健康检查结果 |
| `GhostLinkEntry` / `GhostLinkPreview` | 幽灵链接条目 + 预览 |
| `CleanupResult` / `MigrationStats` | 清理结果 + 迁移统计 |

### 3. `utils.rs` — 工具函数（126 行）

**职责**: 跨模块共享的纯函数，无状态无副作用。

| 函数 | 说明 |
|------|------|
| `is_junction(path)` | 检测路径是否为 NTFS Junction（FILE_ATTRIBUTE_REPARSE_POINT） |
| `get_junction_target(path)` | 读取 Junction 的目标路径（去除 `\\?\` 前缀） |
| `get_folder_size(path)` | 递归计算文件夹大小（fs_extra::get_size） |
| `expand_env_vars(path_str)` | 展开 `%APPDATA%`、`%USERPROFILE%` 等环境变量 |
| `check_disk_space_for_restore(target_dir, size)` | 还原前检查目标盘剩余空间（要求 ≥ 1.1× 文件大小） |
| `app_data_templates_path(data_dir)` | 获取 `app_data_templates.json` 路径 |
| `custom_folders_path(data_dir)` | 获取 `custom_folders.json` 路径 |
| `history_file_path(data_dir)` | 获取 `migration_history.json` 路径 |

### 4. `system/` — 系统接口层

#### `system/disk_usage.rs`（63 行）

**职责**: 使用 `sysinfo` crate 枚举所有磁盘，返回容量/可用/使用率。系统盘优先排序。

**对外接口**: `get_disk_usage() -> Result<Vec<DiskUsage>, String>` — Tauri 命令

#### `system/icon.rs`（222 行）

**职责**: 使用 Win32 API 从 `.exe`/`.dll` 中提取图标并编码为 `data:image/png;base64,...`。

**技术流程**: `ExtractIconExW` → `GetIconInfo` → `GetDIBits`（BGRA 像素数据） → BGRA→RGBA 转换 → `image` crate 编码 PNG → `base64` crate 编码

**性能优化**: 使用 `lazy_static!` + `Mutex<HashMap>` 实现图标缓存，避免重复提取同一图标。

**对外接口**: `extract_icon_to_base64(icon_path: &str) -> String`

### 5. `storage/` — 存储层

#### `storage/data_dir.rs`（136 行）

**职责**: 管理 OrbitFile 的数据目录。

**架构**:
- 指针文件 `%APPDATA%/orbit-file.json`（几十字节）→ 记录实际数据目录路径
- 默认数据目录 `%APPDATA%/orbit-file/`
- 支持运行时切换数据目录，自动迁移数据文件
- 原子写入（临时文件 → sync → rename）防止断电损坏

**对外接口**:
| 函数 | 类型 | 说明 |
|------|------|------|
| `get_data_dir()` | 内部 | 读取指针文件获取实际数据目录 |
| `ensure_data_dir()` | 内部 | 确保数据目录存在（缺失则自动创建） |
| `get_data_dir_info()` | Tauri 命令 | 返回当前数据目录路径（供设置页展示） |
| `set_data_dir(path)` | Tauri 命令 | 修改数据目录 + 迁移数据文件 |
| `load_custom_folders(path)` | 内部 | 读取自定义文件夹列表 |
| `save_custom_folders(path, folders)` | 内部 | 保存自定义文件夹列表 |

#### `storage/history.rs`（473 行）

**职责**: 迁移历史记录的完整生命周期管理。

**持久化方案**: JSON 文件（`migration_history.json`），选择 JSON 而非 SQLite 的原因：
1. 轻量级 — 无额外依赖
2. 可读性 — 用户可直接查看
3. 低频写入 — JSON 完全够用

**对外接口**:
| 函数 | 类型 | 说明 |
|------|------|------|
| `load_history()` / `save_history()` | 内部 | 读写 JSON 历史文件（原子写入策略） |
| `add_migration_record(...)` | 内部 | 迁移成功后追加记录 |
| `update_migration_record_status(...)` | 内部 | 按路径更新记录状态 |
| `get_migration_history()` | Tauri 命令 | 返回活跃的迁移记录 |
| `get_migrated_paths()` | Tauri 命令 | 返回所有已迁移路径列表 |
| `check_link_status(record_id)` | Tauri 命令 | 检查 Junction 链接健康状态 |
| `preview_ghost_links()` | Tauri 命令 | 扫描目标路径不存在的记录（只读） |
| `clean_ghost_links()` | Tauri 命令 | 删除失效 Junction + 标记记录 |
| `get_migration_stats()` | Tauri 命令 | 统计已节省空间、活跃/已恢复数量 |
| `export_history(path)` | Tauri 命令 | 导出历史文件到指定路径 |
| `import_history(path)` | Tauri 命令 | 从文件导入并合并记录（按 ID 去重） |
| `restore_app(history_id)` | Tauri 命令 | **6 步还原流程**（见下方详细说明） |
| `open_data_dir()` | Tauri 命令 | 在资源管理器中打开数据目录 |
| `open_folder(path)` | Tauri 命令 | 在资源管理器中打开指定文件夹 |

**`restore_app` 6 步还原流程**:
1. 查找迁移记录（按 ID + active 状态）
2. 验证状态（目标路径存在 + 原路径是 Junction）
3. 空间检查（目标盘有足够空间，含 10% 缓冲）
4. 删除 Junction（`fs::remove_dir`）
5. 移动文件夹回原位置（`fs_extra::move_dir`，失败时重建 Junction 回滚）
6. 更新记录状态为 "restored"

### 6. `folder_manager/` — 文件夹管理层（416 行）

**职责**: 系统文件夹、应用数据文件夹、自定义文件夹的发现、迁移和恢复。

**文件夹来源**:
| 类型 | 来源 | 示例 |
|------|------|------|
| 系统文件夹 | `dirs` crate（Windows Shell API） | 桌面、文档、下载、图片、视频 |
| 应用数据 | `app_data_templates.json` + `detector` 模块 | 微信、QQ、钉钉、飞书、Chrome 缓存等 |
| 自定义 | 用户手动添加 → `custom_folders.json` | 任意文件夹 |

**核心流程**: `get_large_folders` 返回列表 → 后台 `compute_folder_sizes_async` 异步计算大小 → 通过事件推送前端

**迁移/恢复**: 耗时 IO 在独立线程执行，完成后通过事件通知前端（不阻塞 UI 线程）

**模板热更新**: `app_data_templates.json` 可运行时编辑，无需重启应用

### 7. `app_manager/` — 应用管理层

#### `app_manager/scanner.rs`（607 行）

**职责**: 扫描 Windows 已安装应用。

**扫描策略（三层回退）**:
| 层级 | 数据源 | 覆盖场景 |
|------|--------|----------|
| 1. 注册表 | HKLM/HKCU Uninstall 键（× 32/64-bit） | 标准安装器 |
| 2. DisplayIcon 回退 | 从 DisplayIcon/UninstallString 推导安装路径 | ComfyUI 等便携安装器 |
| 3. 文件系统扫描 | Program Files / Program Files (x86) / LocalAppData | Squirrel 安装器（Electron 应用） |

**去重策略**: 按规范化路径去重，注册表条目优先于文件系统条目。

**对外接口**: `get_installed_apps() -> Result<Vec<InstalledApp>, String>`（Tauri 命令）

#### `app_manager/migration.rs`（330 行）

**职责**: 应用迁移引擎。

**9 步迁移流程**:
1. 空间检查（目标盘可用空间 ≥ 源大小）
2. 进度上报（通过 `migration-progress` 事件实时推送）
3. 逐个复制（遍历源目录，按文件数上报百分比）
4. 完整性校验（比较源和目标文件夹总大小）
5. 备份原目录（重命名为 `xxx_backup`）
6. 创建 Junction（`symlink_dir` 在原位置创建目录联接）
7. 写入迁移历史（调用 `storage::history::add_migration_record`）
8. 清理备份（删除 `xxx_backup`）
9. 回滚机制（任何步骤失败尝试恢复原状态）

**取消支持**: 每个步骤前检查 `cancel_flag`，用户可随时中断。

#### `app_manager/uninstaller.rs`（2159 行）

**职责**: 应用卸载引擎，对齐 Geek Uninstaller 标准流程。

**卸载流程**:
1. 读取注册表 UninstallString（原始命令）
2. 运行卸载向导（用户正常交互）
3. 等待卸载进程退出（权限不足时通过 PowerShell 提权）
4. 确认注册表条目已移除
5. 残留扫描（`%APPDATA%` / `%LOCALAPPDATA%` / `ProgramData` + 注册表）
6. 用户确认后清理（支持回收站/彻底删除两种模式）

**安全机制**:
| 安全层 | 说明 |
|--------|------|
| `BLACKLIST` | 禁止卸载/删除系统关键应用（Microsoft、Windows 等） |
| `validate_deletion_target` | 4 层校验防止误删系统文件 |
| `StrictScanContext` | 多维度匹配（应用名/文件夹名/发布商/安装路径） |
| 回收站模式 | 删除文件先放入回收站（在设置中可切换） |

#### `app_manager/detector.rs`（299 行）

**职责**: 动态检测聊天/办公类应用的数据目录（微信、QQ、钉钉、飞书等）。

**检测方式**: 遍历各应用的已知存储路径模式，结合注册表信息，确认实际数据目录位置。

**对外接口**:
- `get_special_folders_status() -> Result<Vec<FolderStatus>, String>` — 返回所有特殊文件夹的检测状态
- `migrate_special_folder(...)` — 含进程预检的安全迁移入口

#### `app_manager/log_macros.rs`（14 行）

**职责**: 统一后端日志格式 `[orbit-file][级别][模块] 消息`。

## 关键设计决策

| 决策 | 理由 |
|------|------|
| JSON 而非 SQLite 存储历史 | 轻量、可读、低频写入场景足够 |
| 迁移/恢复耗时操作放入独立线程 | 避免阻塞 Tauri 线程池，通过事件异步通知前端 |
| 原子写入（temp → sync → rename） | 防止写入过程中断/崩溃损坏数据 |
| 图标缓存（lazy_static + Mutex） | 避免重复提取同一图标，每图仅提取一次 |
| `models.rs` 集中管理所有类型 | 单一真相来源，避免类型定义分散在多处 |
| `utils.rs` 零依赖基础模块 | 确保所有模块都可安全引用，无循环依赖风险 |
| `storage/history.rs` 中放置 `open_folder` 等杂项命令 | 与 `restore_app`、`open_data_dir` 同属文件操作范畴 |
