# Viap 后端模块说明

## 架构总览

```
src-tauri/src/
│
├── main.rs                 # 程序入口（6 行）— 仅调用 viap_lib::run()
├── lib.rs                  # 模块枢纽（210 行）— 声明子模块 + 注册 35 个 Tauri 命令
│
├── models.rs               # 【基础层】共享数据结构 — 所有模块的零依赖上游
├── utils.rs                # 【基础层】文件系统工具 + 全局恢复锁
│
├── system/                 # 【系统接口层】Windows OS 信息查询
│   ├── mod.rs              #   模块声明（5 行）
│   ├── disk_usage.rs       #   磁盘容量/使用率扫描（sysinfo）
│   └── icon.rs             #   从 .exe/.dll 提取图标 → Base64 PNG（Win32 API）
│
├── storage/                # 【存储层】数据目录 / 迁移历史 / 元数据 / 操作日志
│   ├── mod.rs              #   模块声明（7 行）
│   ├── data_dir.rs         #   数据目录配置（指针文件、自定义文件夹 CRUD）
│   ├── history.rs          #   迁移历史 CRUD、幽灵链接/无效记录扫描与清理、统计、导入导出、应用还原
│   ├── migrated_app_metadata.rs  # 迁移应用元数据兜底（绿色软件不丢失）
│   └── operation_log.rs    #   操作日志（卸载/强删/清理审计轨迹，FIFO 轮转）
│
├── folder_manager/         # 【文件夹管理层】系统 / 应用数据 / 自定义文件夹的迁移
│   └── mod.rs              #   大文件夹发现 + 异步大小扫描 + 迁移/恢复 + 模板管理
│
└── app_manager/            # 【应用管理层】Windows 已安装应用的全生命周期
    ├── mod.rs              #   模块声明（10 行）
    ├── cache.rs            #   内存级应用缓存（RwLock + 脏标记 + 图标复用 + 兜底补全）
    ├── scanner.rs          #   注册表扫描（HKLM/HKCU × 32/64-bit）+ 文件系统补充扫描
    ├── migration.rs        #   迁移引擎（空间检查 → 复制 → 校验 → Junction → 清理备份）
    ├── uninstaller.rs      #   卸载引擎（原始命令执行 → 注册表确认 → 残留扫描 → 安全清理）
    ├── detector.rs         #   特殊目录检测（聊天应用数据路径动态发现）
    └── log_macros.rs       #   统一日志宏（orbit_log! / log_warn!）
```

## 模块依赖关系

```
models.rs  ←────────────────────────────────────────── 所有模块
utils.rs   ←────────────────── storage/  folder_manager/  app_manager/
    ↑                              ↑           ↑               ↑
    │                              │           │               │
system/                      storage/    folder_manager/  app_manager/
(独立,仅依赖     (data_dir 为上游,           (依赖 storage/        (内部互调,
 models)          history / metadata           + app_manager/)     依赖 storage/)
                  / op_log 并列)
```

- **`models.rs`** 和 **`utils.rs`** 零上游依赖，被所有其他模块引用
- **`system/`** 独立，仅依赖 `models`，供 `app_manager/` 图标提取使用
- **`storage/`** 四个子模块并列：
  - `data_dir.rs` 是上游（提供 `ensure_data_dir()`）
  - `history.rs` 依赖 `data_dir.rs`（读写 JSON 历史文件）和 `migrated_app_metadata.rs`（恢复/清理时间接清理元数据）
  - `migrated_app_metadata.rs` 依赖 `data_dir.rs` + `system/icon.rs`（兜底应用列表需提取图标）
  - `operation_log.rs` 依赖 `data_dir.rs`
- **`folder_manager/`** 依赖 `storage/`（读历史）、`app_manager/detector.rs`（路径检测）、`app_manager/migration.rs`（复用迁移引擎）
- **`app_manager/`** 内部：`migration.rs` 依赖 `storage/history.rs`（写迁移记录）和 `storage/migrated_app_metadata.rs`（写兜底元数据）；`scanner.rs` 依赖 `system/icon.rs`；`cache.rs` 依赖 `scanner.rs` + `storage/migrated_app_metadata.rs`
- **`lib.rs`** 负责将所有命令桥接到 Tauri，`migrate_app` / `migrate_special_folder` 两个耗时命令在 `lib.rs` 中封装 `spawn_blocking` 并注入 `MigrationState` 取消标志

---

## 各模块详细说明

### 1. `main.rs` — 程序入口（6 行）

```rust
fn main() { viap_lib::run() }
```

仅设置 `windows_subsystem = "windows"` 并委托给 `lib::run()`。

### 2. `lib.rs` — 模块枢纽（210 行）

**职责**: 声明 6 个子模块，注册 35 个 Tauri 命令，启动应用。

**命令注册方式**：
- **直接转发**（14 个命令）：`get_installed_apps`、`check_process_locks` 等 → 委托给 `app_manager/`
- **带状态注入 + spawn_blocking**（2 个命令）：`migrate_app`、`migrate_special_folder` → 注入 `MigrationState` 取消标志，耗时 IO 在独立线程执行
- **模块路径直连**（19 个命令）：`generate_handler![]` 中直接引用子模块的 `#[tauri::command]` 函数

### 3. `models.rs` — 共享数据结构（271 行）

**职责**: 集中定义所有前后端共享的类型，零业务逻辑，序列化/反序列化唯一真相来源。

| 结构体 | 用途 |
|--------|------|
| `MigrationState` | Tauri 托管状态，持有取消标志 `Arc<AtomicBool>` |
| `InstalledApp` | 已安装应用（名称、路径、图标 Base64/URL、发布商、注册表路径） |
| `DiskUsage` | 磁盘容量 / 可用 / 使用率 / 系统盘标识 |
| `LargeFolder` / `LargeFolderType` | 大文件夹信息 + 类型枚举（System / AppData / Custom） |
| `LargeFolderSizeEvent` | 后台异步计算大小后的事件载荷 |
| `DataDirConfig` / `CustomFolderEntry` | 数据目录配置 + 自定义文件夹持久化条目 |
| `AppDataTemplate` | 应用数据模板（定义监控哪些应用的数据目录） |
| `MigrationResult` / `ProcessLockResult` | 迁移结果 + 进程占用检测结果 |
| `MigrationRecord` / `MigrationRecordType` / `HistoryStorage` | 迁移历史记录 + 类型枚举 + 持久化容器 |
| `LinkStatusResult` | Junction 链接健康检查结果 |
| `GhostLinkEntry` / `GhostLinkPreview` | 无效记录条目（含 damage_type）+ 预览 |
| `CleanupResult` / `MigrationStats` | 清理结果 + 迁移统计 |

### 4. `utils.rs` — 工具函数 + 全局锁（160 行）

**职责**: 跨模块共享的纯函数 + 全局恢复并发锁。

| 符号 | 说明 |
|------|------|
| `is_junction(path)` | 检测路径是否为 NTFS Reparse Point（FILE_ATTRIBUTE_REPARSE_POINT 位检测） |
| `get_junction_target(path)` | 读取 Junction 的目标路径（去除 `\\?\` 前缀） |
| `get_dir_size_safe(path)` | 权限无关的目录大小计算（WalkDir 遍历，跳过无权限条目）|
| `get_folder_size(path)` | 获取文件夹大小（兼容旧接口，委托 get_dir_size_safe） |
| `expand_env_vars(path_str)` | 展开 `%APPDATA%`、`%USERPROFILE%` 等环境变量 |
| `check_disk_space_for_restore(target_dir, size)` | 还原前检查目标盘剩余空间（要求 ≥ 1.1× 文件大小）|
| `app_data_templates_path(data_dir)` | 获取 `app_data_templates.json` 路径 |
| `custom_folders_path(data_dir)` | 获取 `custom_folders.json` 路径 |
| `history_file_path(data_dir)` | 获取 `migration_history.json` 路径 |
| `RESTORE_LOCK` | 全局 `AtomicBool` 恢复互斥锁，防止并发恢复任务互相干扰 |
| `RestoreLockGuard` | RAII 锁守卫，`Drop` 时自动释放锁，保证函数异常路径也安全 |
| `try_acquire_restore_lock()` | CAS 获取锁，返回 `Result<RestoreLockGuard, String>` |

### 5. `system/` — 系统接口层

#### `system/disk_usage.rs`（63 行）

**职责**: 使用 `sysinfo` crate 枚举所有磁盘，返回容量/可用/使用率。系统盘优先排序。

**对外接口**: `get_disk_usage() -> Result<Vec<DiskUsage>, String>`（Tauri 命令）

#### `system/icon.rs`（248 行）

**职责**: 使用 Win32 API 从 `.exe`/`.dll` 提取图标并编码为 `data:image/png;base64,...`。

**技术流程**: `ExtractIconExW` → `GetIconInfo` → `GetDIBits`（BGRA 像素数据）→ BGRA→RGBA 转换 → `image` crate 编码 PNG → `base64` crate 编码。

**对外接口**: `extract_icon_to_base64(icon_path: &str) -> String`

### 6. `storage/` — 存储层

#### `storage/data_dir.rs`（136 行）

**职责**: 管理 Viap 的数据目录。

**架构**:
- 指针文件 `%APPDATA%/viap.json` → 记录实际数据目录路径
- 默认数据目录 `%APPDATA%/viap/`
- 支持运行时切换数据目录，自动迁移数据文件
- 原子写入（临时文件 → sync → rename）防止断电损坏

**对外接口**:

| 函数 | 类型 | 说明 |
|------|------|------|
| `get_data_dir()` | 内部 | 读取指针文件获取实际数据目录 |
| `ensure_data_dir()` | 内部 | 确保数据目录存在（缺失则自动创建） |
| `get_data_dir_info()` | Tauri 命令 | 返回当前数据目录路径 |
| `set_data_dir(path)` | Tauri 命令 | 修改数据目录 + 迁移数据文件 |
| `load_custom_folders(path)` | 内部 | 读取自定义文件夹列表 |
| `save_custom_folders(path, folders)` | 内部 | 保存自定义文件夹列表 |

#### `storage/history.rs`（539 行）

**职责**: 迁移历史记录的完整生命周期管理。

**持久化方案**: JSON 文件 `migration_history.json`，选择 JSON 的原因：
1. 轻量级 — 无额外依赖
2. 可读性 — 用户可直接查看
3. 低频写入 — JSON 完全够用

**对外接口**:

| 函数 | 类型 | 说明 |
|------|------|------|
| `load_history()` / `save_history()` | 内部 | 读写 JSON 历史文件（原子写入策略） |
| `add_migration_record(...)` | 内部 | 迁移成功后追加记录 |
| `update_migration_record_status(...)` | 内部 | 按路径更新记录状态 |
| `get_migration_history()` | Tauri 命令 | 返回活跃的迁移记录 |
| `get_migrated_paths()` | Tauri 命令 | 返回所有已迁移路径列表 |
| `check_link_status(record_id)` | Tauri 命令 | 检查 Junction 链接健康状态 |
| `preview_ghost_links()` | Tauri 命令 | 扫描无效记录（三类损坏，只读） |
| `clean_ghost_links()` | Tauri 命令 | 清理无效记录（删除 Junction + 标记状态） |
| `get_migration_stats()` | Tauri 命令 | 统计已节省空间、活跃/已恢复数量 |
| `export_history(path)` | Tauri 命令 | 导出历史文件 |
| `import_history(path)` | Tauri 命令 | 导入并合并记录（按 ID 去重） |
| `restore_app(history_id)` | Tauri 命令 | 7 步还原流程（见下方） |
| `open_data_dir()` | Tauri 命令 | 在资源管理器中打开数据目录 |
| `open_folder(path)` | Tauri 命令 | 在资源管理器中打开指定文件夹 |

**`restore_app` 7 步还原流程**:
0. 获取全局恢复锁（防止并发）
1. 查找迁移记录（按 ID + active 状态）
2. 验证状态（目标路径存在 + 原路径是 Reparse Point）— 使用 `utils::is_junction` 而非 `symlink_metadata().is_symlink()`，因为 Windows Junction 在 Rust 标准库中 `is_symlink()` 返回 false
3. 空间检查（目标盘有足够空间，含 10% 缓冲）
4. 删除 Junction（仅确认是 Reparse Point 时才删除）
5. 移动文件夹回原位置（`fs_extra::move_dir`，失败时重建 Junction 回滚）
6. 更新记录状态为 "restored" + 移除兜底元数据

**`preview_ghost_links` / `clean_ghost_links` 三类损坏检测**:

| 类型 | 判断条件 | 清理行为 |
|------|----------|----------|
| `target_missing` | 目标路径不存在 | 删除 Junction（如存在）→ 更新状态 |
| `junction_broken` | 原路径存在但不是 Reparse Point | 拒绝删除（保护数据）→ 仍更新状态 |
| `original_missing` | 原路径不存在且不是 Reparse Point | 无需清理 → 直接更新状态 |

#### `storage/migrated_app_metadata.rs`（172 行）

**职责**: 绿色软件兜底机制。绿色软件无注册表条目，迁移后其安装目录变为 Junction，扫描器可能遗漏。迁移成功时将元数据写入 `migrated_apps.json`，应用列表加载时读取补全。

**对外接口**:

| 函数 | 类型 | 说明 |
|------|------|------|
| `add_migrated_app(...)` | 内部 | 迁移成功后新增元数据条目（去重） |
| `remove_migrated_app(path)` | 内部 | 恢复/清理后移除元数据条目 |
| `generate_failsafe_apps(existing)` | 内部 | 生成兜底应用列表（仅补全扫描遗漏的路径） |

#### `storage/operation_log.rs`（135 行）

**职责**: 操作日志审计。记录卸载、强制删除、残留清理的完整轨迹。`uninstall_logs.json` 存储，FIFO 轮转（最多 100 条），新记录在列表头部。

**对外接口**:

| 函数 | 类型 | 说明 |
|------|------|------|
| `add_operation_log(...)` | 内部 | 追加操作日志（自动 FIFO 轮转） |
| `get_operation_logs()` | Tauri 命令 | 获取全部日志（倒序，最新在前） |

### 7. `folder_manager/` — 文件夹管理层（460 行）

**职责**: 系统文件夹、应用数据文件夹、自定义文件夹的发现、迁移和恢复。

**文件夹来源**:

| 类型 | 来源 | 示例 |
|------|------|------|
| 系统文件夹 | `dirs` crate（Windows Shell API） | 桌面、文档、下载、图片、视频 |
| 应用数据 | `app_data_templates.json` + `detector` 模块 | 微信、QQ、钉钉、飞书、Chrome 缓存等 |
| 自定义 | 用户手动添加 → `custom_folders.json` | 任意文件夹 |

**核心流程**: `get_large_folders` 返回列表（size=0）→ 前端注册 `large-folder-size` 监听器 → 调用 `start_folder_size_scan` 触发后台异步计算 → 通过事件推送大小到前端。分离扫描与计算是为了消除竞态（后台线程在前端监听器就绪前 emit 事件会导致事件丢失）。

**迁移**: 复用 `app_manager::migration::migrate_app` 引擎，通过 `migration-progress` 事件实时上报进度，支持 `cancel_migration` 取消。

**恢复**: `async + spawn_blocking`，先获取全局恢复锁（与 `restore_app` 互斥），再执行 Junction 删除 + 目录移回。与 `history.rs` 共用 `utils::RESTORE_LOCK`。

**模板热更新**: `app_data_templates.json` 可运行时编辑，无需重启应用。

### 8. `app_manager/` — 应用管理层

#### `app_manager/cache.rs`（131 行）

**职责**: 内存级应用缓存，避免 Tab 切换或重复操作时触发全量扫描。

**核心机制**:
- `RwLock<AppCache>` 全局单例（`lazy_static`）
- 脏标记 `is_dirty` 驱动：缓存有效时直接返回内存数据，失效时触发全量扫描
- 图标复用：扫描前提取旧缓存的 (路径, 图标) → (Base64, URL) 快照，新扫描结果中路径不变的直接复用旧图标，避免重复提取
- 兜底补全：扫描完成后调用 `migrated_app_metadata::generate_failsafe_apps` 补全遗漏的已迁移绿色软件
- 迁移后 `invalidate` 缓存，卸载后 `retain` 移除

**对外接口**:

| 函数 | 类型 | 说明 |
|------|------|------|
| `get_or_scan()` | Tauri 命令（经 lib.rs） | 获取应用列表（缓存命中则直接返回） |
| `refresh()` | Tauri 命令（经 lib.rs） | 强制刷新缓存 |
| `on_app_migrated(...)` | 内部 | 迁移后标记缓存脏 |
| `on_app_uninstalled(...)` | 内部 | 卸载后从缓存移除 |

#### `app_manager/scanner.rs`（1610 行）

**职责**: 扫描 Windows 已安装应用。

**扫描策略（三层回退）**:

| 层级 | 数据源 | 覆盖场景 |
|------|--------|----------|
| 1. 注册表 | HKLM/HKCU Uninstall 键（× 32/64-bit） | 标准安装器 |
| 2. DisplayIcon 回退 | 从 DisplayIcon/UninstallString 推导安装路径 | 便携安装器 |
| 3. 文件系统扫描 | Program Files / Program Files (x86) / LocalAppData | Squirrel 安装器（Electron 应用） |

**去重策略**: sort + linear scan，O(n log n)，注册表条目优先于文件系统条目。

**对外接口**:

| 函数 | 类型 | 说明 |
|------|------|------|
| `scan_all()` | 内部 | 执行全量三层扫描 |
| `get_installed_apps()` | Tauri 命令 | 经由 cache 层获取应用列表 |
| `get_app_size(path)` | Tauri 命令 | 计算指定目录大小 |
| `check_process_locks(path)` | Tauri 命令 | 检测路径是否被运行中进程占用 |

#### `app_manager/migration.rs`（464 行）

**职责**: 应用迁移引擎。

**迁移流程**:
1. 空间检查（目标盘可用空间 ≥ 源大小，通过最长前缀匹配确定挂载盘）
2. 逐个复制（遍历源目录，按文件数上报百分比，支持取消）
3. 完整性校验（比较源和目标总大小，允许跳过无权限文件 + 1MB 容差）
4. 备份原目录（重命名为 `xxx_backup`）
5. 创建 Junction（`symlink_dir` 在原位置创建目录联接）
6. 写入迁移历史（`storage::history::add_migration_record`）
7. 写入兜底元数据（`storage::migrated_app_metadata::add_migrated_app`）
8. 清理备份（删除 `xxx_backup`）

**取消支持**: 每个步骤前检查 `cancel_flag`，取消时清理目标目录避免残留。

**对外接口**: `migrate_app(...)` — 内部函数，供 `lib.rs`（应用迁移）和 `folder_manager/`（文件夹迁移）复用。

#### `app_manager/uninstaller.rs`（2134 行）

**职责**: 应用卸载引擎，对齐 Geek Uninstaller 标准流程。

**卸载流程**:
1. 读取注册表 UninstallString（原始命令）
2. 运行卸载向导（用户正常交互）
3. 等待卸载进程退出（权限不足时通过 PowerShell 提权）
4. 确认注册表条目已移除
5. 残留扫描（`%APPDATA%` / `%LOCALAPPDATA%` / `ProgramData` + 注册表）
6. 用户确认后清理（支持回收站/彻底删除两种模式）

**安全机制**:

| 安全层 | 说明 |
|--------|------|
| `BLACKLIST` | 禁止卸载系统关键应用 |
| `validate_deletion_target` | 4 层校验防止误删系统文件 |
| `StrictScanContext` | 多维度匹配（应用名 / 文件夹名 / 发布商 / 安装路径）|
| 回收站模式 | 删除文件先放入回收站（设置中可切换） |

#### `app_manager/detector.rs`（311 行）

**职责**: 动态检测聊天/办公类应用的数据目录（微信、QQ、钉钉、飞书等），并支持含进程预检的安全迁移。

**检测方式**: 遍历各应用的已知存储路径模式，结合注册表信息，确认实际数据目录位置。npm 全局包优先执行 `npm config get prefix` 获取用户自定义路径。

**对外接口**:
- `get_special_folders_status()` → 返回所有特殊文件夹的检测状态
- `migrate_special_folder(...)` → 含进程预检的安全迁移入口

#### `app_manager/log_macros.rs`（14 行）

**职责**: 统一后端日志格式 `[viap][级别][模块] 消息`。通过 `#[macro_use]` 在 `mod.rs` 中引入，`app_manager` 内部所有子模块可使用 `orbit_log!` / `log_warn!`。

## 关键设计决策

| 决策 | 理由 |
|------|------|
| JSON 而非 SQLite 存储 | 轻量、可读、低频写入场景足够 |
| 耗时操作 spawn_blocking | 避免阻塞 Tauri IPC 线程，通过事件异步通知前端 |
| 原子写入（temp → sync → rename）| 防止写入过程中断/崩溃损坏数据 |
| 图标缓存（内存级 RwLock）| 避免重复提取同一图标，迁移/卸载后增量更新 |
| `models.rs` 集中管理所有类型 | 单一真相来源，避免类型定义分散 |
| `utils.rs` 零依赖基础模块 | 确保所有模块都可安全引用，无循环依赖风险 |
| 全局恢复锁 `RESTORE_LOCK` | 前端 UI 锁（restoringId 单值 state）无法阻止快速并发 invoke，必须后端原子锁兜底 |
| `is_junction` 检测 REPARSE_POINT 属性位 | Windows Junction 在 Rust 标准库 `is_symlink()` 返回 false，只有 REPARSE_POINT 位能准确识别 |
| 文件夹大小扫描与列表获取分离 | 消除竞态：前端在 `large-folder-size` 监听器就绪后才调用 `start_folder_size_scan`，避免事件丢失 |
| 迁移应用元数据兜底 | 绿色软件无注册表条目，迁移后可能从扫描器遗漏，独立 JSON 文件补全 |
| 缓存层脏标记驱动 | 迁移/卸载后仅标记失效，下一轮访问才扫描，避免连锁触发 O(n) 全量操作 |
